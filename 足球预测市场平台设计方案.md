# 足球预测市场平台（FootballPredict）设计方案

> 类 Polymarket 的去中心化体育（足球）预测市场平台 —— 系统设计文档 v1.0
> 适用范围：MVP 架构选型、核心模块设计、安全风控与扩展路线图

---

## 0. 项目定位与产品形态

平台目标是构建一个面向足球赛事的去中心化预测市场（Prediction Market）。用户可以围绕「英超切尔西 vs 阿森纳，下半场谁先进球」「世界杯小组赛 X 队是否晋级」等结果型问题进行交易，由市场价格反映群体概率预期，结果出炉后由 Oracle 喂入链上完成自动结算。

与传统博彩的核心差别在于：赔率不是由庄家设定，而是由用户买卖 YES/NO 份额（Outcome Token）形成的市场价格决定，平台只赚取手续费、不与用户对赌。这一机制借鉴自 Polymarket / Augur / Gnosis CTF（Conditional Tokens Framework）。

平台 MVP 形态建议为「链下撮合 + 链上结算」的混合架构：交易撮合在中心化引擎上完成（追求性能与体验），资金托管、份额铸造与最终结算在链上完成（追求可信与抗审查）。这是 Polymarket 当前主流做法。

## 1. 核心模块设计

### 1.1 市场与事件管理（Market & Event Management）

**职责**：负责足球比赛元数据维护、预测市场创建、生命周期管理。

**数据模型（核心实体）**：

- `Sport / League / Season`：体育门类、联赛、赛季。如「英超 2026-2027」。
- `Match`：单场比赛，包含主客队、开球时间、场地、轮次、状态机（未开始 / 进行中 / 已结束 / 已取消 / 已推迟）。
- `Market`：预测市场。一个 Match 可以挂载多个 Market（胜平负、大小球 2.5、首先进球队、半场比分、角球数等）。
- `Outcome`：单个市场下的可选结果（YES/NO 二元，或多元如 主胜/平局/客胜）。
- `MarketRules`：结算规则文本与机器可读字段（如「全场结束的比分判定」「点球大战是否计入」）。
- `MarketLifecycle`：包含 开盘时间、停止下单时间（一般为比赛开球或开球前 5 分钟）、预计结算时间、争议窗口期。

**市场创建流程**：

1. 运营或社区提案人通过后台/治理合约提交市场参数（Match ID、市场类型、结算规则、初始流动性）。
2. 风控审核（自动 + 人工）：检查规则是否清晰可裁决、是否有可信 Oracle 数据源。
3. 系统调用智能合约 `MarketFactory.createMarket()` 部署/注册新市场，铸造 YES/NO（或多 Outcome）份额合约（基于 ERC-1155 的 Conditional Tokens 模式更省 gas）。
4. 注入初始流动性（LMSR 或 CPMM 做市商，见交易系统章节）。
5. 市场进入「Open」状态，前端展示。

**生命周期状态机**：

`Draft → PendingReview → Open → Locked（停止下单）→ Reporting（Oracle 上报中）→ Disputable（争议窗口）→ Resolved → Settled / Voided`

任何状态切换都应当上链存证，且发出事件供前端与索引器订阅。

### 1.2 交易系统（Trading & Matching）

**两种主流模式可选，建议采用混合方案**：

**方案 A：AMM 做市（CPMM / LMSR）**
- 适合冷门市场和长尾合约，保证任何时刻都有可成交流动性。
- LMSR（Logarithmic Market Scoring Rule）专为预测市场设计，价格自然落在 [0,1] 区间，可以直接解读为概率，做市商最大损失有界。
- CPMM（恒定乘积）实现简单，但价格滑点在极端概率下偏大。

**方案 B：CLOB 链下订单簿 + 链上结算**
- Polymarket 当前主路径。订单在链下撮合引擎排队和撮合，成交后由 Exchange 合约用 EIP-712 签名订单批量上链结算。
- 优点：低 gas、高吞吐、用户体验接近 CEX；缺点：撮合是中心化点，需要透明化处理。

**推荐：MVP 用 AMM 起步，热门赛事市场逐步引入 CLOB**，并在前端统一抽象为「买 YES 0.62 USDC」这种用户视角。

**赔率与价格**：
- 二元市场：YES 价格 P_YES ∈ (0,1)，NO 价格 = 1 - P_YES，对应隐含概率与十进制赔率 1 / P_YES。
- 多元市场：所有 Outcome 价格之和恒等于 1 USDC（结算资产），任何一份完整 Outcome 集可在结算前以 1 USDC 兑换。
- 价格更新由订单成交或 AMM 流动性变化驱动，链下推送给前端（WebSocket）。

**下单/撮合逻辑（CLOB 版本）**：

1. 用户在前端选择 YES/NO 与数量，调用钱包对订单做 EIP-712 签名（包含 marketId、outcome、price、size、nonce、expiry）。
2. 撮合引擎收到签名订单，写入订单簿（按价格-时间优先），并对家撮合。
3. 撮合成功后产生「成交意图」，由 Operator 批量打包提交到 `Exchange` 合约，合约校验签名、转移 USDC、铸造/转移 Outcome Token。
4. 部分成交保留剩余仓位，未成交订单可由用户主动取消（撤单上链签名失效或链下标记）。
5. 提供做市商激励，鼓励专业 MM 提供深度。

**保证金与杠杆**：MVP 不引入杠杆，全部全额下单。后期可基于 Outcome Token 做 LP 挖矿与抵押借贷。

### 1.3 用户账户与钱包管理

平台用户账户体系建议采用「Web2 登录 + 链上自托管 + 智能账户（AA）」组合：

- **登录方式**：邮箱/社交登录（Web3Auth、Privy、Magic）一键生成嵌入式钱包；同时支持 MetaMask、Rabby、WalletConnect 等浏览器钱包；移动端支持 Coinbase Wallet 等。
- **智能合约钱包（ERC-4337）**：让新用户无需先持有 Gas 即可下单，使用 Paymaster 由平台代付 gas，再从下单金额中扣除等价 USDC，极大降低 Web3 门槛。Polymarket 就是这一思路。
- **多链托管**：MVP 选择单链（Polygon PoS 或 Base），日后多链可通过 LayerZero / CCIP 桥接 USDC 与仓位。
- **KYC/合规**：根据上线地区决定。海外版可保持无 KYC，但需要地理封锁（美国等）；面向更合规市场可对接 Sumsub / Persona。
- **资金账户分层**：链上「自托管钱包」（用户主权）→「Exchange 合约托管」（撮合期间锁仓）→「Outcome Token 余额」。所有操作都通过签名授权，平台不掌握私钥。

### 1.4 结算机制（Settlement）

**结算流程**：

1. 比赛结束 → Match 状态切到 `Finished`，主数据源（如 Sportradar / Opta）回传最终比分。
2. Oracle 模块（见 1.5）将结果上报到链上 `OracleAdapter` 合约：例如 `setOutcome(marketId, winningOutcomeId)`。
3. 进入「争议窗口」（建议 30 分钟 - 6 小时，热门赛事可缩短）。期间任何持仓用户都可质押保证金发起争议，触发 UMA Optimistic Oracle / Kleros 仲裁。
4. 争议窗口结束无异议 → `Resolution` 合约调用 `payout`，把 USDC 按胜出 Outcome Token 持仓 1:1 兑付。
5. 用户在前端点击「领取奖金」（claim）签名上链，资金到账。也可由平台批量推送（gasless）。

**部分结算与特殊情况**：

- **比赛取消/延期**：市场进入 `Voided`，所有持仓按购买成本退还（1:1 USDC 退款）。
- **平局/未到达条件**：按规则文本，可能 YES 与 NO 均无效（Void）或全部归 NO。
- **数据源分歧**：进入仲裁，等待最终判决。
- **极端情况（如 VAR 改判）**：以官方比赛报告为最终依据，规则需事先在 MarketRules 中写明。

**结算资产**：MVP 推荐 USDC（Circle 原生 USDC，避免桥接资产风险），后期可加入 USDT、DAI、ETH。

### 1.5 数据获取与 Oracle 设计

足球数据源选择：

- **商业 API**：Sportradar、Opta（Stats Perform）、SportMonks、API-Football，提供高频实时数据与历史回填。
- **公共/半免费 API**：football-data.org、TheSportsDB（可靠性较低，作为辅助校验）。
- **链上专用 Oracle**：Chainlink Sports Data Feeds、UMA Optimistic Oracle、Pyth Network。

**Oracle 架构（多源 + 乐观）**：

平台采用「多源喂入 + 乐观确认 + 仲裁兜底」的三层结构：

1. **采集层**：后端定时（赛中 5-30 秒一次，赛后立刻）拉取 2-3 个独立 API 数据，做一致性比对。
2. **共识层**：当 N 选 M（如 3 选 2）数据一致，标记为「Provisional Result」，将哈希写入链上 `OracleAdapter` 并发起 `proposeOutcome`。
3. **仲裁层**：使用 UMA Optimistic Oracle 模型——任何人在争议窗口内可质押挑战。无人挑战即生效；被挑战则进入 UMA DVM 或 Kleros 仲裁，由代币持有者投票决定真实结果，错误挑战方被罚没保证金。
4. **熔断机制**：当数据源出现重大偏差（比分跳变、状态异常）自动暂停市场结算，触发人工运维。

这样可以避免「单点 Oracle 喂错数据导致大额错误结算」的灾难，是预测市场最关键的安全设计之一。

## 2. 技术栈建议

### 2.1 区块链与智能合约

- **链选择**：MVP 选 **Polygon PoS** 或 **Base**（低 gas、EVM 兼容、Circle 原生 USDC 支持、用户基础好）。长期可扩展至 Arbitrum、Optimism、zkSync。
- **合约语言**：**Solidity 0.8.24+**，配合 Foundry（推荐）或 Hardhat 测试。
- **核心合约模块**：
  - `ConditionalTokens`（基于 Gnosis CTF，ERC-1155 标准的 Outcome Token）
  - `MarketFactory` / `MarketRegistry`
  - `Exchange`（CLOB 结算）/ `AMM`（LMSR 或 CPMM）
  - `OracleAdapter`（适配 UMA / Chainlink）
  - `PaymentProcessor`（USDC 充值/提现/手续费分配）
  - `Governance`（治理与参数升级，可用 OpenZeppelin Governor）
  - `Treasury`（平台收入与保险基金）
- **依赖库**：OpenZeppelin Contracts、Gnosis Conditional Tokens、UMA Optimistic Oracle V3、Permit2（gasless 授权）、Solady（gas 优化版基础库）。

### 2.2 后端服务

- **语言/框架**：**Node.js + TypeScript（NestJS）** 或 **Go（Gin/Fiber）**。
  - NestJS 适合业务逻辑复杂、模块化清晰的场景，社区丰富。
  - Go 适合撮合引擎、行情推送等性能敏感模块。
  - 推荐：**业务网关 NestJS + 撮合引擎 Go/Rust**（撮合用 Rust 性能更优，参考 Polymarket 用的 Go/Rust 自研引擎）。
- **数据库**：
  - **PostgreSQL**（主库，存储用户、市场、订单、成交、结算记录）
  - **Redis**（订单簿热缓存、行情快照、限流、分布式锁）
  - **ClickHouse / TimescaleDB**（行情 K 线、成交明细、用户行为分析）
  - **The Graph / Subgraph** 或自建索引器（Goldsky、Ponder）订阅链上事件，作为链上→链下的标准数据通道。
- **消息队列**：**Kafka** 或 **NATS**，用于撮合事件、行情广播、Oracle 任务、结算任务的异步分发。
- **任务调度**：Temporal（推荐，工作流容错强）或 BullMQ。
- **实时推送**：**WebSocket（Socket.IO 或 uWebSockets）** 推送行情、订单状态、结算通知。

### 2.3 前端

- **框架**：**Next.js 14+（App Router） + React 18 + TypeScript**。
- **UI**：Tailwind CSS + shadcn/ui，行情图用 TradingView Lightweight Charts 或 ECharts。
- **钱包集成**：
  - **wagmi + viem**（推荐，类型安全、现代化）
  - **RainbowKit / ConnectKit / Reown AppKit (WalletConnect v2)** 提供钱包选择 UI
  - **Privy / Web3Auth** 邮箱登录 + 嵌入式钱包
  - **Account Abstraction（ERC-4337）**：ZeroDev、Pimlico、Biconomy SDK，实现 Gasless 与一键下单。
- **状态管理**：Zustand 或 React Query（TanStack Query）+ Jotai。
- **移动端**：React Native（Expo）或 PWA 双轨。

### 2.4 基础设施与运维

- **部署**：Kubernetes（EKS/GKE）+ Helm + ArgoCD；或更轻量的 Railway / Fly.io（MVP 阶段）。
- **节点服务**：Alchemy / Infura / QuickNode + 自建归档节点冗余。
- **监控**：Prometheus + Grafana + Loki；链上监控 Tenderly、Defender（OpenZeppelin）。
- **CDN/边缘**：Cloudflare（防 DDoS + 边缘缓存静态行情）。
- **日志与审计**：所有撮合、结算、Oracle 调用全链路 trace（OpenTelemetry）。

### 3.2 核心数据流与资金流

**下单数据流（CLOB 模式）**：

`用户前端` → 选择 YES @ 0.62、size=100 → `wagmi 签名 EIP-712 订单` → `API Gateway` → `Matching Engine（Rust，内存订单簿）` → 撮合命中对手单 → 写 Kafka `trade.executed` → 三路并行：(1) `Quote Service` 广播 WebSocket 新成交与新最优买卖一价；(2) `Settlement Worker` 把成交批量发到 `Exchange` 合约（aggregate transfer USDC + Outcome Token）；(3) `PostgreSQL` 入账历史成交。

**资金流**：

- 充值：用户从外部钱包 `transfer USDC → Exchange 合约托管子账户`（或直接持仓在自己钱包，使用 Permit2 单签授权）。
- 下单锁仓：买 YES @ 0.62 × 100 = 62 USDC 被锁定，撮合成功后转移给卖方，买方收到 100 单位 YES Token。
- 结算兑付：比赛结果 = YES 胜出 → 100 YES Token 可在 `Resolution` 合约 1:1 兑换 100 USDC。
- 提现：用户从自托管钱包直接转出，无平台审批环节。
- 平台手续费：在 `Exchange` 撮合时按成交额 0.5%-2% 抽取至 `Treasury`，部分注入「保险基金」用于异常结算赔付。

**Oracle 数据流**：

`Sportradar/Opta 多源采集` → `Oracle Aggregator（M of N 一致性校验）` → `propose 上链` → `争议窗口（UMA OO）` → 无争议则 `resolve` → 触发 `Settlement Worker` 推送结算 → 用户领取。

### 3.3 高并发与缓存优化

足球比赛存在「赛前 5 分钟 + 进球瞬间」的瞬时流量尖峰（QPS 可达平时 50-100 倍），架构需重点针对此优化：

- **撮合引擎**：单市场内串行（保证撮合一致性），不同市场水平分片（按 marketId hash），单机内存订单簿 + WAL 持久化，故障切换 < 1 秒。
- **行情广播**：Redis Pub/Sub + WebSocket fan-out，热点市场采用「快照 + 增量」推送，前端使用 Throttle 减少渲染。
- **读写分离**：PostgreSQL 一主多从，重读场景（用户持仓、历史订单）路由到只读副本；高频写场景（订单簿、成交）首先入 Redis / Kafka，再异步落库。
- **多级缓存**：CDN 缓存静态市场列表（短 TTL，60s） → Redis 缓存赔率快照（1-3s） → 进程内 LRU 缓存最热点数据（毫秒级）。
- **链上写入降本**：成交批量打包（Batch Settlement），Roll-up 思路把 N 笔成交在一个交易内完成，gas 摊薄。
- **限流与降级**：Token Bucket 限流；当链上拥堵时自动延长撮合提交窗口；启用排队页防止雪崩。
- **冷热分离**：已结算市场归档至 ClickHouse / S3，活跃市场保留在 OLTP 库。

## 4. 安全与风控设计

### 4.1 智能合约安全

- **重入攻击**：所有外部调用使用 OpenZeppelin `ReentrancyGuard` 保护，遵循 Checks-Effects-Interactions 模式；优先使用 `transfer` / `call` 时严格控制 gas。
- **整数溢出/下溢**：Solidity 0.8+ 默认开启检查，再针对关键累加用 `unchecked` 与 `SafeCast` 显式管控。
- **签名重放**：订单使用 EIP-712 + nonce + chainId + expiry，撮合合约维护 nonce bitmap，Permit2 替代 approve 减少前置授权风险。
- **权限边界**：核心合约使用 `AccessControl` 分角色（OPERATOR、ADMIN、ORACLE、GOVERNANCE），Admin 上 Timelock + Multisig（Safe），避免单点跑路。
- **升级路径**：UUPS / Transparent Proxy，升级走 Timelock 7 天延迟 + 公开提案，赋予用户退出时间。
- **审计与测试**：Foundry 单元测试 + 模糊测试（Echidna）+ 形式化验证（Certora，可选）+ 第三方审计（Trail of Bits / OpenZeppelin / Sigma Prime）+ Bug Bounty（Immunefi）。

### 4.2 防市场操纵

- **流动性深度门槛**：低于阈值的市场限制大额单笔成交，防止价格瞬时拉抬。
- **价格波动熔断**：单市场单时间窗内价格漂移超过 X%（如 30% / 60s）时暂停撮合 N 秒。
- **女巫识别**：钱包行为聚类（同一资金源、同一 IP、相似下单模式），结合 Gitcoin Passport / WorldID 做信誉评分。
- **赛前/赛中信息差**：临开球前 N 分钟禁止大额下单（参数化），降低内幕信息利用空间。
- **做市商激励**：白名单 MM 提供保证深度，平抑剧烈波动；同时监控 MM 自成交。

### 4.3 Oracle 与结算异常处理

- **多源喂入 + Optimistic Oracle**：见 1.5。
- **争议保证金**：发起争议需质押结算金额的 5%-10%，错误争议罚没，避免恶意挑战。
- **熔断与人工接管**：当三大数据源出现两两不一致或异常停跳，系统进入 `Frozen` 状态，提案进入治理流程。
- **保险基金**：手续费的一部分注入 `Treasury → Insurance Pool`，覆盖极端情况下因 Oracle 错误或合约漏洞导致的用户损失。
- **赛事取消/异常**：写明在 MarketRules 中，状态切到 `Voided` 全额退款，避免事后争议。

### 4.4 用户与合规风控

- **地理封锁**：根据地区法规对受限国家用户做 IP/钱包筛查（Chainalysis Oracle、TRM Labs）。
- **反洗钱**：大额异常充值/提现风控规则，必要时触发增强审查。
- **反钓鱼**：官方域名 EV 证书 + Etherscan 验证合约源码 + 前端事务前展示「Simulation」（Blowfish / Tenderly Simulation API）。

## 5. 可选增强功能

### 5.1 批量市场结算（Batch Settlement）
单场比赛通常对应 10-50 个市场（胜平负、大小球、角球、进球队等），结算时使用 `Multicall` 一次性把同场所有市场的 `resolve + payout` 打包，节省 80%+ gas。

### 5.2 AI 预测提示（Insights）
- 接入 LLM（如 Claude / GPT-4）+ 历史数据（球队 ELO、xG、伤病、主客场战绩、天气），为每个市场生成「AI 概率参考 + 简短解读」。
- 明确标注为「研究性参考，非投注建议」，并展示模型最近 N 场的命中率与校准度（Brier Score），透明展示模型靠谱程度。
- 进一步可让用户订阅特定球队/联赛的 AI 提醒。

### 5.3 多币种下注
- 一期主推 USDC，二期扩展 USDT、DAI、ETH、平台代币。
- 多币种统一以 USDC 计价撮合，通过 Uniswap V3 / 1inch 自动换汇（设置滑点保护），用户体验上「我用 ETH 下注」。
- 平台代币（FBM Token）持有可减免手续费、参与争议仲裁投票、分享 Treasury 收入。

### 5.4 历史统计与用户排行榜
- 用户画像：累计盈亏（PnL）、命中率、平均赔率、最大回撤、活跃天数。
- 排行榜：日榜/周榜/赛季榜，按 PnL、命中率、交易量等维度，前列用户可获得 NFT 徽章与代币空投。
- 跟单（Copy Trading）：可选关注「带头大哥」并按比例自动跟单，撮合时附带跟单标记，便于审计与限制以防操纵。
- 社交分享：成交 / 大胜单可一键生成 OG 图片分享 X / Telegram，自带邀请码，做增长闭环。

### 5.5 其他增强方向
- **赛中（In-Play）市场**：进球瞬间秒级开/合盘，对撮合引擎与 Oracle 实时性要求极高。
- **组合市场（Parlay）**：多个 Outcome 组合下单，链上用 CTF 的 Compound Position 表达。
- **Mobile Push & Telegram Bot**：开盘、临开球、结算通知，并直接在 TG 内下单（深度链接到 mini-app）。
- **DAO 治理**：参数（手续费率、争议窗口、Treasury 用途）通过治理代币投票决定。

## 7. 关键风险提示（开发前必须知道）

1. **合规风险**：预测市场在多个司法辖区被认定为博彩，需要法律意见与地理封锁，建议先以离岸实体运营。
2. **Oracle 是头号风险**：80% 的预测市场事故出在 Oracle，多源 + 乐观仲裁是底线。
3. **撮合中心化点**：链下撮合必须发布每日审计快照、订单簿哈希上链，否则用户对「撮合公平」无可验证手段。
4. **流动性冷启动**：早期市场没有用户对手盘，必须有自营/合作 MM 注入流动性，并准备激励预算（流动性挖矿）。
5. **MEV 与抢跑**：链上结算环节注意防抢跑（私有 Mempool / Flashbots Protect）。
